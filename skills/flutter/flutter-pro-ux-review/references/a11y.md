### Touch targets large enough

`min-touch-target` ¬∑ Critical on primary actions; Noticeable on secondary ¬∑ Effort: S  
Fix in one phrase: interactive targets ‚â• ~48√ó48 dp (padding counts); don‚Äôt rely on 16dp icons alone.

**Detect:** `IconButton` with tiny constraints, custom `GestureDetector` icons without `minimumSize` / padding, dense chips that are the only control for a primary action.

**Hunt:** icon-only actions in app bars and lists; check `constraints`, `padding`, `minimumSize` on `ButtonStyle`.

**Why:** mis-taps and exclusion of users with motor difficulty.

**Gotchas:** dense data tables may use smaller secondary targets if a larger hit area still exists; document exceptions.

### Semantics on icon-only and image buttons

`semantics-tappable` ¬∑ Critical on primary; Noticeable on secondary ¬∑ Effort: S  
Fix in one phrase: every tappable without visible text gets a `Semantics` label / `tooltip` / `IconButton` semantic label.

**Detect:** `IconButton` / `InkWell` / `GestureDetector` with only an `Icon` and no `tooltip`, `semanticsLabel`, or wrapping `Semantics(label:, button: true)`.

**Hunt:** grep `IconButton|Icons\.` in onTap widgets; check for labels.

**Gotchas:** decorative non-interactive icons should be excluded via `ExcludeSemantics` / `Semantics(excludeSemantics: true)` ‚Äî don‚Äôt flag those as missing labels if they aren‚Äôt tappable.

### Text scale must not destroy layout

`text-scale-survival` ¬∑ Critical if primary UI overflows/clips at large scale; Noticeable if cramped ¬∑ Effort: M  
Fix in one phrase: layouts survive large text (flexible text, ellipsis/scroll, or intentional clamp with care).

**Detect:** fixed-height rows with multi-line labels, `Text` in tight `Row`s without `Flexible`/`Expanded`, hardcoded heights that clip at 1.3‚Äì2.0 text scale.

**Hunt:** dense list tiles, bottom nav labels, buttons with fixed height; grep `height:` near `Text`/`ListTile`.

**Why:** system text size is an a11y setting; clipped primary labels are blockers.

**Gotchas:** intentional `textScaler` clamp (e.g. max 1.2) can be OK for complex tools if documented; still flag broken primary auth/home at default large sizes.

### Status not by color alone

`status-not-color-only` ¬∑ Noticeable ¬∑ Effort: S  
Fix in one phrase: success/error/warning include icon or text, not only red/green.

**Detect:** status rows that change only `color:` / container color with no icon/text change.

**Hunt:** error/success banners, validation UI, order status chips.

**Why:** color-blind and low-vision users miss the state.

**Gotchas:** brand accents on already-labeled text are fine.
